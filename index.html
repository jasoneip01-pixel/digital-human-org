数字人组织

Enterprise AI Collaboration OS
企业 AI 协作操作系统 — 让每个员工拥有 10 个数字实习生

产品定义说明书 · 商业模式设计 · 2026.07 · v1.1 架构深化

01 产品定位与愿景

让企业像管理员工一样管理 AI 数字员工 — v1.1 架构深化版

一句话定义： 数字人组织 = 一个企业级的 AI Agent 管理与编排平台。它在个人 AI 助手（如 WorkBuddy）的基础上增加了 「组织」层——支持多 Agent 的岗位定义、权限管理、流程协作、绩效监控和合规审计。

愿景： 成为企业 AI 协作的操作系统层。核心洞察——AI 在企业中的最佳角色是协作伙伴（数字同事），而非替代者。每个员工配 10 个数字实习生，执行 Propose→Review→Execute→Verify 闭环。

v1.1 定位跃迁： 从「数字员工管理平台」→「企业 AI 协作操作系统」。三个新增核心支柱：① Agent 间通信协议（IACP） ② @agent 标签路由 ③ 跨层安全基础。

不是做什么

  • 不是又一个 AI 对话界面
  • 不是低代码 RPA 平台
  • 不是大模型 API 网关
AI Agent 引擎 企业组织管理 工作流编排

02 核心问题

企业在部署 AI Agent 时遇到的系统性矛盾

🔥 管理黑洞

AI 项目散落在各部门——HR 不知谁负责，IT 不知权限怎么设，老板不知 ROI 怎么算。

🔥 治理真空

50 个 Agent 同时运行时：谁对输出负责？数据能不能跨域？决策怎么留痕？

🔥 集成地狱

Agent 要调 CRM/ERP/OA/企微，不做集成层，Agent 只在聊天框里回答问题。

🔥 规模失效

1 个 Agent POC 简单，1→100 个要解决角色/权限/知识库隔离/成本分摊/性能监控。

06A 第一部分：Inter-Agent Protocol（IACP）NEW

Agent 间通信协议 — 数字人组织的「神经系统」

缺失影响： 没有 Agent 间通信协议 → 所有 Agent 只能通过编排层中介通信 → 单点瓶颈 + 灵活度受限。企业 100+ Agent 场景下编排层成为不可接受的吞吐瓶颈。

协议四原语

📡 Request-Response

Agent A 向 Agent B 请求数据/操作，等待响应。同步模式，适合查询、验证。

📢 Event Pub/Sub

Agent 发布事件（如「发票已审核」），订阅者自动接收。异步模式，适合通知、日志。

🔄 Task Delegation

Agent A 将子任务委派给 Agent B/C，跟踪结果。适合工作流分解。

✅ Result Acknowledgment

接收方返回结构化回执：{status, changeset, errors, confidence}。

⚡ Conflict Resolution

两个 Agent 产出冲突时，采用 last-write-wins、CRDT 或向上级 Agency 仲裁。

通信拓扑

模式适用场景优势劣势
P2P 直连固定协作对（财务↔法务）低延迟，无中介耦合度高
Mediated（中介路由）动态任务分发灵活，可审计性能开销
Orchestrator复杂多步骤流程上下文统一单点风险
Mesh探索型任务（研究/分析）无单点，自组织审计困难

06B 第二部分：@agent 标签路由系统 NEW

借鉴 Claude Code @claude 标签 → 企业 IM 消息流中的 @agent 路由

灵感来源： Claude Code 的 @claude 标签允许用户通过注释标记代码文件来调用 Claude。在企业 IM（飞书/企微/钉钉）中，@agent(角色) 标签可以天然嵌入消息流。

// Claude Code 模式
@claude Review this function for SQL injection vulnerabilities

// → 数字人组织映射
@agent(财务) 核对本月供应商发票，标记异常项
@agent(法务) 审查这个合同的合规风险
@agent(客服) 处理这批工单，所有退款需 @human 审批

标签体系

标签类型语法示例
领域标签/finance /hr /legal /tech /sales@agent(/finance) 核对应收账款
动作标签/query /execute /approve /review@agent(/execute) 批量导入客户数据
优先级标签/urgent /routine /scheduled@agent(/urgent) 处理异常交易预警
安全标签/confidential /internal /public@agent(/confidential) 审查并购文件
协作标签@human /@team /@approver@agent(/legal) 起草合同，@approver 审批

路由规则

🎯 Capability-based

根据 Agent 注册的能力自动匹配：/finance/execute → 财务数字人

🏷️ Tag-based

用户显式指定：@agent(财务) 强制路由

⚖️ Load-aware

同类 Agent 有多个时，按负载和优先级分发（类似任务队列）

🔀 Context-aware

根据对话上下文推断（群聊历史→检测到财务话题→自动建议财务 Agent）

06C 第三部分：Cowork 模式 & Human-in-the-Loop NEW

从「AI 替代人」到「人+AI 协作」— Claude Code 协作模式的企业级迁移

Claude Code → 数字人组织直接映射

Claude Code 模式→ 数字人组织映射企业场景
@claude 文件级标签@agent(角色) 任务级路由标签群聊中 @agent(法务) 审查合同
/approve 审批机制岗位审批链（role-based approval）财务 10 万以下自动、以上需 CFO 审批
multi-step planning工作流 Step/Phase 自动分解入职流程：HR→IT→财务 三步自动编排
diff preview执行前 changeset 预览Agent 修改数据库前展示变更清单
terminal shell execTool Use 沙箱执行SAP 参数由 Agent 填写，人工 confirm 执行
session persistence任务记忆 + 岗位记忆Agent 记住每个员工的偏好和工作习惯

四种 Human-in-the-Loop 模式

🔐 Pre-approval

AI 提案 → 人工审批 → AI 执行
适用：法务审查、大额支付、对外发布内容

📊 Post-audit

AI 执行 → 记录日志 → 人工抽样复核
适用：客服回复、数据标注、报表生成

🚨 Escalation

AI 执行 → 异常触发上报 → 人工介入
适用：风控预警、系统异常、置信度<阈值

📅 Review

AI 自动 → 定期批量人工复核
适用：费用报销、考勤异常、合规扫描

推销话术重构： 不说「这个 AI 能替你的 3 个人」，说「每个人配 10 个数字实习生，你把关就好」。前者引发抗拒，后者引发好奇。

06D 第四部分：多 Agent 编排模式 NEW

四种编排模式共存 — 不同场景走不同拓扑

核心洞察： 持久化 Agent Identity（岗位定义）和按需 Task-oriented Instance（临时实例）可以共存——同一个「客服岗数字人」在不同时间以 Orchestrator / Worker / Supervisor 切换。Identity ≠ Instance

🎭 Orchestrator/Plugin

一个编排 Agent 接收任务 → 拆解 → 分配给专用 Sub-Agent → 汇总结果

默认模式

🔗 Sequential Chain

Agent A → B → C 流水线传递
例：发票识别→核对→审批→支付→归档

审批流程

🕸️ Mesh/Swarm

Agent 间自由协调，无中央控制
例：竞品分析、创意生成、研究

探索

👁️ Supervisor/Worker

Supervisor 分派→多 Worker 并行→汇总
例：批量客户工单、数据清洗

批量

编排选择矩阵

场景推荐模式理由
HR 入职流程（多部门串联）Sequential ChainHR→IT→财务，有先后依赖
客服工单（批量处理）Supervisor/Worker大量工单并行，汇总结果
竞品分析（无固定流程）Mesh/Swarm多 Agent 从不同角度分析，自由碰撞
月度报表（多数据源）Orchestrator/Plugin中央调度，聚合多源数据
合同审批（多层级审批）Sequential Chain + HIL每步有审批节点

06E 第五部分：记忆架构 NEW

四层记忆体系 — 从单次对话到组织知识

💬 对话记忆

当前会话上下文
生命周期：会话结束即释放
技术：LLM context window

📋 任务记忆

跨会话的任务持久化
生命周期：任务完成 → 归档
技术：向量数据库 + 摘要

🧠 岗位记忆

持续学习的岗位经验
生命周期：持续积累
技术：偏好模型 + 个人知识库

🏢 组织知识库

全局共享的企业知识
生命周期：永久
技术：RAG + KG + 权限隔离

关键设计原则： ① 记忆隔离 — 法务数字人不能读到财务数字人的对话记忆 ② 向上收敛 — 有价值的对话记忆 → 提炼为任务记忆 → 沉淀为岗位记忆 → 上升到组织知识 ③ 遗忘曲线 — 超过 90 天未访问的记忆自动归档，降低存储成本

06F 第六部分：安全基础 NEW

跨层安全基础 — 企业采购的前置条件

AI 特有攻击面： ① Prompt 注入（用户输入恶意指令控制 Agent）② Agent 间侧信道（Agent A 输出含敏感数据，Agent B 在日志中读取）③ 工具滥用（Agent 获得权限后执行未授权操作）④ 数据泄露（Agent 记忆中含 PII / 商业机密）

🛡️ Prompt 注入防护

输入过滤器（检测 jailbreak pattern）+ 输出过滤器（脱敏重写）+ 沙箱验证

P0

🔑 RBAC/ABAC

角色权限（RBAC）+ 属性权限（ABAC，如「仅在 9-18 时访问财务系统」）

P0

📜 Immutable Audit

Agent 所有决策链不可篡改记录——谁输入、AI 推理了什么、做了什么、谁审批的

P1

🔒 Agent Namespace 隔离

每个数字人运行在独立沙箱/容器，跨数字人数据访问需显式授权

P1

🕵️ Data Classification

数据分级（公开/内部/机密/绝密）→ 自动限制低级别 Agent 的数据访问范围

P2

06G 第七部分：生态协同 & 能力复用 NEW

数字人组织 × agent-docs 工具链的协同矩阵

已有能力协同方式优先级
MCP Registry · MCP 工具注册表作为数字人组织的默认工具源——所有 Agent 通过 MCP Registry 发现和安装工具P0
Agent Cost CLI · 推理成本追踪作为成本中心数据源——每个数字人的推理费用自动上报到 Cost CLIP1
Agent DataDog · 跨框架可观测性作为 Observability Layer 的基础——分布式追踪 + 仪表盘 + 告警P1
Skill Scanner · Agent 能力审计审计每个数字人的技能版本——检测能力变更、安全漏洞、过期依赖P2
Skill Surface Diff · 能力变更分析追踪数字人岗位定义变更——谁改了哪个数字人的技能、什么时候改的P2

协同基线： 当前 v1.0 产品定义完全没有引用这些已有能力——每个都各自独立。v1.1 应明确数字人组织与 agent-docs 工具链的集成路径，避免重复造轮子。
